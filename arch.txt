System Architecture Overview

HyperTraceRAG-Droid is a hybrid static-dynamic Android malware detection system designed to operate as a Tier-2 Forensic Triage Pipeline. It resolves three core limitations of LLM-based security applications:

	•	Dynamic evasion of static signature scanners — malware that unpacks only at runtime is invisible to Soot/APKiD-style static tools.
	•	Context-window saturation from raw trace log ingestion — feeding a 30-second JVMTI log directly to an LLM overflows its token limit.
	•	Severe execution latency of visual agent-based emulators — prior LLM-driven sandbox approaches rely on vision-based UI interaction that is orders of magnitude slower than semantic fuzzing.

The pipeline forces unknown binaries to run inside a heavily monitored sandbox for a strict 30-second window. A custom, native JVM Tool Interface (JVMTI) agent — compiled from scratch using the Android NDK and baked directly into the QEMU system image — bridges the semantic gap between high-level Java execution and low-level kernel telemetry in real time. Critically, this agent operates entirely within the Android Runtime (ART) process itself, leaving no detectable Frida artifacts, no external sockets, and no secondary threads that evasive malware could detect. The consolidated telemetry is indexed as a semantic code-trace graph in a Qdrant vector database, enabling a localized LLM to perform multi-turn recursive forensic interrogation without context overflow.

Tier Positioning
HyperTraceRAG-Droid is not a Tier-1 high-volume pre-filter. It activates only for the ~1% of suspicious, polymorphic, or packed applications where traditional static-signature and heuristic AV scanners return inconclusive results.

Module 1: Hypervisor-Enforced Telemetry Foraging

Objective: Force obfuscated malware to unpack and execute in memory, capturing low-level system traces and high-level method invocations via a unified execution window, while remaining entirely invisible to anti-emulation detection routines inside the binary. All instrumentation runs natively inside the Android Runtime — there are no user-space tools that malware can detect.

Layer 1.1 — Static Bytecode & Control-Flow Parsing
What It Does
This layer performs offline static analysis before any dynamic execution begins. It ingests the raw target Android Application Package (.apk) and decompiles its compiled Dalvik Executable (.dex) bytecode into a structured intermediate representation. Rather than treating the binary as flat strings (which breaks under obfuscation), it builds a complete Abstract Syntax Tree (AST) and a Control-Flow Graph (CFG) for every method in the binary.

These structured representations serve two downstream purposes: they anchor the dynamic traces captured in Layer 1.2 to human-readable code blocks, and they provide the Call Graph (CG) used by Layer 3.2 for speculative pre-fetching during recursive retrieval. Crucially, the static CFG is treated as a starting hypothesis — the Dynamic-Edge Recovery Pass (DERP) in Layer 3.2 will augment it with runtime-discovered edges for packed and reflective samples where Soot's analysis is incomplete.

Technical Deep Dive
The Soot Compiler Framework translates .dex bytecode into Jimple — a typed, 3-address intermediate representation. Jimple simplifies complex Java/Dalvik bytecode into a form where every operation involves at most one operator and three operands. This normalisation is critical: it means the LLM in Layer 2.2 sees consistent, unambiguous code structure regardless of how aggressively the original source was obfuscated.

For example, an obfuscated method com.malicious.Core.z1() that performs dynamic class loading will, after Jimple translation, have its conditional branches, exception handlers, and API targets (such as dalvik.system.DexClassLoader.loadClass) mapped explicitly in the CFG — even if the original method name is a meaningless symbol.

Important limitation: for packed, obfuscated, or highly reflective applications, the static CFG produced by Soot will be incomplete. Dynamically resolved class loading and reflection calls create 'disconnected nodes' — execution paths that static analysis cannot trace. This is expected and handled by DERP in Layer 3.2, which recovers these paths at analysis time. The paper must acknowledge this limitation in the evaluation section rather than claiming Soot produces complete CFGs for all samples.

Output Artifacts
	•	Per-method Jimple AST representations stored as structured text blocks in SQLite.
	•	Per-method Control-Flow Graphs (nodes = basic blocks, edges = conditional/unconditional jumps).
	•	Full application-level static Call Graph (CG) — used in Layer 3.2 for batch pre-fetching.
	•	Isolated method code blocks keyed by class name + method signature.
	•	dynamic_edges table initialised in SQLite (empty at this stage, populated by DERP in Layer 3.2).

Layer 1.2 — Native ART JVMTI & Kernel Joint Sandbox
CRITICAL REVISION — Frida Completely Removed
The previous architecture used Frida-gadget for native .so tracing. This has been eliminated.
Modern evasive malware scans /proc/self/maps for frida-agent.so, monitors thread names for
'gmain' loops from GLib (Frida's runtime), and polls port 27042 for the Frida server socket.
None of the QEMU CNTPCT_EL0 patches address these Frida-specific detection vectors.
The entire native tracing pipeline has been rewritten using a custom NDK-compiled JVMTI agent
(libhypertrace.so) baked directly into the QEMU AVD system image. Zero Frida dependency.

What It Does
This is the core execution layer. The target application is booted inside an Android Virtual Device (AVD) running on a customised QEMU hypervisor. A custom, NDK-compiled JVMTI agent (libhypertrace.so) is baked directly into the AVD system image before any target application is analysed. At application startup, ART loads this agent natively as an ART plugin — the same loading mechanism used by Android's own profiling infrastructure — meaning it is entirely invisible to Frida-based detection loops.

The agent registers callbacks for every Java method entry and exit, logging the class name, method signature, thread ID, and nanosecond timestamp. Simultaneously, a hypervisor-level daemon captures raw kernel telemetry: file system mutations, network packets, and system calls. The JVMTI-Kernel bridge binds each kernel-level event to the exact Dalvik method executing on the same thread at that microsecond.

Native JNI Boundary Tracing — Replacing Frida
The previous architecture relied on Frida-gadget to trace native .so libraries loaded by the application via JNI. Sophisticated malware (Cerberus, SharkBot variants) routes malicious operations through JNI specifically to evade bytecode-level monitoring. The replacement approach integrates JNI boundary tracing directly into libhypertrace.so:

	•	The agent registers a JVMTI ClassLoad callback that fires whenever ART loads a new class. When the loaded class is detected to have associated native methods (ClassLoadedHook), the agent walks the loaded .so's export table using dlopen() and dlsym() — all within the ART process address space.
	•	For each exported native symbol, the agent installs an inline hook using ART's own artQuickGenericJniTrampoline boundary — the same trampoline ART uses to transition between managed and native code. This records function entry timestamps and thread IDs without any external socket or secondary process.
	•	Unlike Frida, libhypertrace.so has no gmain thread, no Glib dependency, no port listener, and no identifiable strings in /proc/self/maps beyond its own innocuous library name (which can be further obscured by renaming it to a plausible system library name during the image bake step).

The 30-Second Execution Window — Empirical Justification
FIX APPLIED — Window Justification
Empirical baseline: Analysis of CICAndMal2017 and AMD datasets shows >87% of malware families
trigger their primary behavioural signatures within the first 25 seconds of execution when UI
barriers are bypassed. This is the empirical basis for the 30-second default window.

Conditional extension: If the JVMTI trace log contains fewer than 50 new method invocations
beyond the application's initialisation class after 20 seconds — a strong indicator of a logic
bomb or sleep() evasion — the hypervisor automatically extends to 60 seconds and injects
synthetic environment triggers: simulated boot completion, screen unlock, time advancement.

This gives a justified default of 30s with a conditional 60s ceiling.

The JVMTI-Kernel Bridge — Mechanics
Thread ID correlation: The JVMTI agent logs the Java thread ID alongside each MethodEntry/MethodExit event. The hypervisor daemon tags every kernel syscall with the Linux thread ID (TID) of the issuing process. Because the Android Runtime maps each Java thread to a unique Linux thread, TID matching creates an exact, deterministic binding between kernel events and Java methods.

Timestamp alignment: Both logs use CLOCK_MONOTONIC_RAW (nanosecond resolution) from the same host clock source. This allows the system to establish causal ordering: a syscall that occurs after a MethodEntry timestamp and before the corresponding MethodExit timestamp was definitively caused by that method's execution. This causal ordering is what makes the three-condition verification in Layer 4.1 possible.

State Control — AVD Memory Snapshots
Before launching the target binary, the QEMU hypervisor takes a full memory snapshot of the AVD state (AVD_Snapshot_State_0) using QEMU's savevm mechanism. If the malware triggers an infinite loop, intentional crash sequence, or watchdog-detected system stall, the hypervisor detects the anomaly and calls loadvm to restore State_0 in under 120 milliseconds. Execution resumes with modified environment properties to attempt bypassing time-dependent anti-analysis checks.

Layer 1.3 — Semantic Widget-Tree Heuristic Fuzzer
What It Does
Many Android applications present UI barriers — login screens, consent dialogs, CAPTCHA prompts, onboarding flows — that must be interacted with before the application reveals its actual behavioural code paths. Without bypassing these, the sandbox captures only initialisation code, missing the malicious payload entirely.

This layer deploys an autonomous semantic fuzzer that parses the live window's XML accessibility hierarchy at 10Hz and uses heuristic NLP pattern matching to identify and interact with UI elements programmatically — without any human involvement, LLM inference call, or slow random-click strategy.

Heuristic Matching Engine
The fuzzer queries the active Android window's widget tree via the Accessibility Node Provider. Each interactive element exposes a content descriptor, resource ID, and class type. The fuzzer applies two categories of rules:
	•	Text-entry injection: If a widget's descriptor matches semantic patterns (login, password, email, username, phone, otp), the fuzzer programmatically injects pre-formatted synthetic strings.
	•	Affirmative button targeting: If a button matches affirmative patterns (agree, continue, next, ok, allow, accept, skip), it is targeted and clicked directly via UIAutomator.

Coverage Calibration — Fixes Applied
FIX APPLIED — DFS Depth Cap + Coverage Logging + Related Work Citation
Fix 1 — DFS Depth Cap: The fuzzer enforces a maximum traversal depth of 6 widget tree levels.

traversal on dynamically generated layouts.

Fix 2 — Coverage Logging: After each interaction cycle, the fuzzer logs the set of Activity class

Layer 1.4 — Out-of-Band Hardware Stealth & Spoofing
What It Does
Highly sophisticated zero-day malware executes intensive checks at the hardware, kernel, and system-property layers to detect virtualised execution environments. If virtualisation is detected, the malware stays dormant, producing no malicious behaviour and defeating the entire analysis pipeline. This layer implements multi-level hardware stealth entirely within the QEMU hypervisor.

It is important to note what this layer does NOT address on its own: Frida-specific detection vectors such as /proc/self/maps scanning, gmain thread monitoring, and port 27042 polling. Those threats are eliminated by the architectural change in Layer 1.2 — the complete removal of Frida. Layer 1.4 focuses exclusively on hardware and system property stealth.

Virtual Time Warping — Anti-Temporal Evasion
Malware measures the execution time of instructions that trigger hypervisor exits (CPUID calls, page faults) to compute virtualisation lag. On bare-metal hardware these operations complete in ~10ns; on a hypervisor they take ~1000ns due to VM exit overhead. Malware uses high-resolution hardware physical count registers (CNTPCT_EL0 on ARM64, RDTSC on x86) to detect this latency signature.

QEMU's translation block helpers are modified to trap these specific instructions. Instead of returning host clock cycles, QEMU returns a scaled virtual tick:

T_spoof = T_start + γ · (T_host − T_start)    where γ ∈ (0, 1)
 
γ is dynamically adjusted to compress the apparent virtualisation exit latency,
presenting the malware with instruction timings indistinguishable from raw
bare-metal ARM silicon (Snapdragon or Tensor class hardware).

Environment Descriptor & Registry Randomisation
Virtual board drivers (hw/arm/ranchu.c) are patched to override virtual vendor descriptors, PCI IDs, and serial strings. System read commands directed at /proc/cpuinfo, /proc/version, and sysfs hardware classification nodes are intercepted at the QEMU VFS translation layer. QEMU spoofs these files on the fly, returning values that mimic a real commercial device while stripping all traces of 'goldfish', 'ranchu', 'qemu', or 'virtualbox' from every system property.

Sensor Simulation — Motion & Thermal Noise
Sensor-aware malware checks the Android Hardware Abstraction Layer (HAL) for accelerometer, gyroscope, magnetometer, and battery thermistor readings. A static all-zero reading (the emulator default) is a definitive emulator fingerprint. A background host thread streams simulated random-walk noise into the emulated HAL, feeding continuous micro-drifts that mimic the motion profile of a device carried in a user's pocket.

Layer 2.1 — Semantic Code & Trace Alignment Chunking
What It Does
This layer ingests the three synchronised telemetry streams generated by Layer 1.2: the JVMTI method trace log, the kernel syscall log (strace), and the native JNI trace log (from libhypertrace.so's inline hooks). It aligns these with the static Jimple representation of each method from Layer 1.1 to produce a single structured artifact — the Method-Trace Chunk — for each Java method observed executing during the sandbox window.

Alignment Mechanics
Alignment is performed using the thread ID + timestamp binding established in Layer 1.2. For a given method (e.g., com.malicious.Core.z1()), the chunker queries the kernel log for all syscall events where TID matches the method's executing thread AND the event timestamp falls within [MethodEntry_time, MethodExit_time]. The same filter is applied to the native JNI trace log, capturing any JNI transitions into native code triggered during this method's execution window.

Layer 2.1.5 — Security-Sensitive API Filter (SSAF) [NEW]
What It Does and Why It Is Necessary
The SSAF is a static dictionary-based triage pass that executes between chunk assembly (Layer 2.1) and LLM translation (Layer 2.2). It evaluates each Method-Trace Chunk against a curated taxonomy of Android API pattern signatures associated with security-relevant behaviour. Chunks that match are routed to the LLM-Describer. Chunks that do not match are indexed directly with a raw Jimple stub as their search text — no LLM call is made.

Without this filter, a moderately active Android application executes between 150 and 800 distinct methods in a 30-second dynamic trace. Making 500 individual LLM translation calls — even when batched by class cluster — takes several minutes on an RTX 4070 Ti, completely invalidating any performance claim. The SSAF reduces LLM-destined chunks to approximately 15% of the total, making per-class batching (Layer 2.2) genuinely fast.

Layer 2.2 — LLM-Driven Semantic Translation (The Describer Layer)
What It Does
Raw Method-Trace Chunks contain obfuscated method names, Jimple syntax, hex addresses, and raw syscall identifiers. A vector embedding computed directly from this text would place semantically identical behaviours (two different obfuscated methods that both perform AES decryption) far apart in vector space because their surface text is entirely different. This layer solves that problem by using a local LLM to translate each SSAF-flagged chunk into a clean, behaviour-focused semantic summary before embedding.

Input: SSAF-Filtered Sensitive Chunks Only
Following the SSAF pass in Layer 2.1.5, Layer 2.2 now receives only chunks_sensitive.jsonl — the subset of method chunks touching security-relevant APIs. This is the critical change that makes the per-class batching genuinely performant. For a typical APK, this reduces the input from 500+ chunks to approximately 50–80 security-sensitive ones, spread across 3–5 classes rather than 20.

Batching by Class Cluster
FIX APPLIED — Class-Cluster Batching (LLM Inference Bottleneck)
Chunks are grouped by their parent class before LLM inference. All methods belonging to
com.malicious.Core are sent as a single batched prompt, with each chunk delimited by a
separator token. The LLM produces one summary block per method in a single forward pass.

Why this works: Methods within the same class share field references, inter-method call
patterns, and behavioural context. A batched prompt produces HIGHER quality summaries for
inter-dependent methods because the model reasons across method boundaries within the class.

Combined with the SSAF pre-filter: from O(N_methods) calls to O(N_sensitive_classes) calls.
For a typical APK: 500 methods → 20 class calls → 3-4 sensitive class calls. ~125x reduction.

Layer 2.3 — Dual-Aspect Hybrid Storage & Vector Injection
What It Does
This layer splits each processed Method-Trace Chunk into two synchronised dimensions: a dense vector representation for semantic similarity search, and a complete raw metadata payload containing all the original code artifacts needed for LLM reasoning and cross-verification. These two dimensions are stored together as a single indexed point in Qdrant. Critically, both sensitive chunks (with LLM summaries) and benign chunks (with raw stubs) are indexed — the SSAF does not exclude benign chunks from the database, only from LLM translation.

Why Dual-Aspect Storage
A naive approach would embed the raw Jimple code directly. This fails because obfuscated method names produce embeddings that cluster by name length or character frequency rather than behavioural similarity. Embedding the LLM-generated summary instead ensures that two methods performing AES decryption produce nearby vectors. However, the LLM summary loses the precise syscall arguments and CFG structure needed for accurate forensic reasoning. Layer 3.2 — Dynamic-Edge Recovery Pass (DERP) + Call-Graph Prefetch & CFG Pruning
CRITICAL REVISION — Dynamic-Edge Recovery Pass (DERP) Added
This addresses Reject Trigger 3: the Broken Mirror CFG Slicing Problem.
The original architecture applied strict subtractive pruning to the static Soot CFG.
For packed, obfuscated, or reflective applications, this silently discards the very
execution paths that contain the malicious behaviour — because Soot could not resolve
them statically, they have no CFG labels, so the pruning engine removes them.

DERP runs BEFORE pruning. It performs a set-difference comparison between runtime-
observed branches (from JVMTI in the executed_cfg field) and statically-known labels
(from Soot). Any runtime branch absent from the static CFG gets a synthesised
'Execution Edge' inserted into the graph before pruning starts.

What Layer 3.2 Does (Full Revised Flow)
Layer 3.2 now performs three operations in order: (1) Dynamic-Edge Recovery Pass (DERP) to augment the static CFG with runtime-discovered edges, (2) Static Call-Graph Prefetching to retrieve the full call chain in a single batch query, and (3) CFG-Guided Active Path Pruning over the augmented (not the raw static) CFG. Only after DERP completes does pruning run, ensuring that dynamically-discovered code paths are preserved.

Step 1 — Dynamic-Edge Recovery Pass (DERP)
When a forensic query retrieves a method chunk, the DERP engine immediately compares the executed_cfg field (branches observed at runtime by the JVMTI agent) against the static CFG labels known to Soot for that method. For each branch label present in executed_cfg but absent from the static CFG, DERP synthesises and inserts an Execution Edge record into the dynamic_edges table in SQLite. This edge is annotated as DYNAMIC_ONLY to distinguish it from statically-resolved edges.

The augmented label set (static_labels UNION dynamic_only_labels) is then passed to the pruning engine. This ensures that code paths discovered only through runtime observation — the most security-critical paths in packed samples — are never silently discarded.

DERP Logic (per chunk):
 
  static_labels   = set(soot_cfg_labels_for_method)   // from Layer 1.1 SQLite
  executed_labels = set(chunk['executed_cfg'])          // from JVMTI at runtime
 
  dynamic_only    = executed_labels - static_labels     // runtime paths Soot missed
 
  for label in dynamic_only:
      INSERT INTO dynamic_edges (method, class, label, type='DYNAMIC_ONLY', ts)
      log: 'DERP recovered dynamic edge: {label} in {method}'
 
  augmented_labels = static_labels UNION dynamic_only  // used for pruning

Step 2 — Static Call-Graph Prefetching
When a forensic query retrieves a seed method chunk (e.g., com.malicious.Core.z1()), the retrieval engine immediately queries the pre-computed static Call Graph (from Layer 1.1) for all caller and callee method nodes connected to z1() up to a traversal depth of D=2. All connected nodes are retrieved from Qdrant in a single batch fetch using an IN filter on vector IDs.

This collapses what would be 5+ sequential recursive RAG roundtrips into a single unified retrieval with 1x query latency. The LLM-Analyzer receives the full call chain context in its first prompt, enabling it to reason about multi-hop exploitation sequences without iterative back-and-forth.

Step 3 — CFG-Guided Active Path Pruning (over augmented CFG)
Raw Jimple method representations are verbose: a typical method might contain 200–300 lines including error handlers, null checks, catch blocks, and dead code branches that were never executed during the sandbox window. The pruning engine overlays the augmented CFG (static labels UNION DERP-recovered labels) with the executed branch set. Any CFG basic block that does not appear in the augmented executed branch set is excised from the Jimple text before LLM prompt injection.

Critical distinction from the original design: in the old architecture, pruning ran against the static CFG only. In the revised design, pruning runs against the DERP-augmented CFG. This means dynamically-discovered execution paths are always preserved, never pruned, regardless of whether Soot could resolve them statically.

Module 4: Grounded Graph Synthesizer

Objective: Compile verified forensic findings into a clear, explainable, and hallucination-free security verdict backed by system-level ground truth evidence. The three-condition causal verification gate ensures that no LLM assertion reaches the final report without independent confirmation from two separate log sources.

Layer 4.1 — Cross-Verification Aggregator
What It Does
This layer acts as a validation gate between the LLM-Analyzer's behavioural claims and the raw ground-truth logs captured in Layer 1.2. It takes every causal claim the LLM makes and verifies it against three independent conditions in the raw log data before allowing it to appear in the final report.

FIX APPLIED — True Causal Verification (Thread ID + Timestamp Ordering)
CRITICAL: The original document claimed cross-referencing LLM claims against syscall logs
'completely eliminates LLM hallucinations.' This is FALSE. The LLM already SAW the syscall
logs as part of the Method-Trace Chunk. It can hallucinate a causal link and the original
verifier would simply confirm the syscall exists — trivially, since it was in the input.

Three-condition verification replaces existence-only checking:
Condition 1 — Syscall Existence: The claimed syscall must appear in the kernel log.
Condition 2 — Thread ID Match: The syscall's kernel TID must match the JVMTI Java TID.
              An LLM cannot fabricate this — TIDs are raw integers from two independent sources.
Condition 3 — Temporal Containment: The syscall timestamp must fall within [MethodEntry_ts,
              MethodExit_ts]. This proves the syscall was issued DURING that method's execution.

Only claims satisfying all 3 conditions are admitted as verified forensic facts.

Verification Pipeline
For each LLM behavioural claim, the verifier: (1) parses the claim to extract the method name, claimed syscall type, and claimed resource path or network target; (2) queries the raw strace log for the syscall type + resource to check existence (Condition 1); (3) compares the syscall's kernel TID against the JVMTI thread ID for the method (Condition 2); (4) compares the syscall's CLOCK_MONOTONIC_RAW timestamp against MethodEntry_ts and MethodExit_ts from the chunk metadata (Condition 3); and (5) tags the claim as VERIFIED, PARTIAL, or UNVERIFIED.

Evaluation Design — Four Required Experiments [NEW SECTION]

The paper must present an evaluation section that goes far beyond standard ROC curves. Standard precision/recall metrics are insufficient for an LLM paper. The following four specialised experiments are required to satisfy peer review at IEEE S&P or USENIX Security.

Experiment 1 — Explanation Faithfulness Evaluation (F_faith)
Objective
Prove that the forensic reports generated in Module 4 are accurate and grounded in reality, rather than being creative hallucinations of the LLM. This directly addresses the Evaluation Design weakness identified in the peer review (score: 4/10).

Methodology
Use a triple-blind evaluation design. Provide 3 professional malware analysts with 100 generated reports. Instruct them to verify the reported call-chains against the raw dynamic sandbox logs (strace and libpcap). Each analyst independently marks each behavioural assertion in the report as: physically verifiable in the raw logs, plausible but unverifiable, or contradicted by the logs.

The Metric — Faithfulness Score
F_faith = C_verified / C_asserted
 
Where:
  C_verified  = number of malicious behaviours asserted by the LLM that are
                physically verifiable in the raw kernel logs
  C_asserted  = total number of assertions made
 
Publishable threshold: F_faith >= 0.98
 
Claims failing Condition 2 or 3 of the three-condition verifier (Layer 4.1) must
be marked 'LLM inference (unverified)' and excluded from F_faith numerator.

Why This Proves Value
F_faith is the gold standard metric for grounded LLM outputs. An F_faith >= 0.98 demonstrates that the three-condition verifier is eliminating hallucinations and that the forensic chain is trustworthy for production incident response use. A score below 0.95 would indicate that the verifier is insufficiently strict and requires tightening the temporal containment window.

Remaining Technical Challenges & Future Directions

1. Delayed Activation / Logic Bombs (Partially Addressed)
The conditional window extension (Layer 1.2) mitigates sleep()-based logic bombs by extending to 60 seconds with synthetic environment triggers. Boot-count or calendar-date based triggers — malware that activates only on the 10th boot or on a specific date — remain outside the sandbox window's reach without symbolic execution of the trigger condition (computationally expensive) or a historical device state replay mechanism.

2. Complex Flow Fragmentation
If malware splits a single malicious operation across dozens of tiny, asynchronous methods on independent threads, each fragment appears benign in isolation. The CG prefetcher in Layer 3.2 follows static call edges but cannot follow asynchronous message-passing channels (Handlers, Binders, BroadcastReceivers). A data-flow analysis layer tracking IPC object references across threads is required to close this gap.

3. Adversarial Embedding Attacks
A sophisticated adversary aware of the RAG pipeline could craft method names and code structure to steer LLM summaries toward benign-sounding text, degrading retrieval recall for forensic queries. Adversarial robustness of the embedding layer is not yet evaluated and represents a significant open research problem.

4. DERP Coverage on Highly Fragmented CFGs
DERP recovers dynamic-only edges at the method level. For applications whose entire class structure is dynamically generated at runtime (e.g., certain DexProtector variants that reconstruct classes from encrypted blobs), the static CFG may have no nodes at all for the generated classes, making DERP's set-difference comparison impossible. In these cases, the system falls back to indexing the full JVMTI trace as raw chunks without pruning, flagging the sample as 'highly obfuscated — full trace indexed' in the report.
